## Use Case

Use these endpoints to build a seat history feature, similar to the one on our [Seats dashboard](https://www.electiondata.my/seats). A user selects a seat by name, and gets back every election result for that seat. Optionally, the user can trace the seat all the way back to the first election, using the seat lineage data provided alongside the results.

The endpoint covers all **222 parliamentary seats** and **600 state seats** that exist today. You can retrieve results in two modes:

- **Default (`lineage=false`)**: results only for elections where the seat held its current name and boundaries. Administratively clean; useful when you want like-for-like comparisons, especially against official classifications.
- **Lineage (`lineage=true`)**: the full chain—results extend back through redelineations and renames, with boundary-change events interspersed to explain the transitions. Use this to recreate the full seat history shown on our site.

Just follow the 2 steps below and you'll be up and running!

## Step 1: Get list of seats

To let users pick a seat, first fetch the full list of current seats and their slugs.

<EndpointBadge method="GET" url="https://api.electiondata.my/v1/seats-current/dropdown" />

This endpoint takes no parameters. The list is only updated when electoral boundaries are changed, so we recommend fetching it once and caching it for the duration of a session (or potentially bundling it at build time for static pages).

With `lineage=true`, the `results` array mixes two object shapes. Election results have the same fields as above. Interspersed between them are **boundary-change events** that mark where the seat transitioned to its dominant ancestor:

<FieldTable fields={[
  { name: "date", type: "date", description: "Rough date of the redelineation event (YYYY-MM-DD); the dates are not exact (i.e. they do not represent the exact date of approval in the Dewan Rakyat), but are sufficiently precise to capture whether a redelineation came into effect before or after an election held in the same year." },
  { name: "change_en", type: "string", description: "English description of the boundary change (e.g. 'Padang Besar was created from Kangar in the 1994 redelineation')." },
  { name: "change_ms", type: "string", description: "Bahasa Malaysia description of the same event (e.g. 'Padang Bsar diwujudkan daripada Kangar dalam persempadanan semula 1994')." },
]} />

You can distinguish the two shapes by checking for the presence of `election_name`: if it exists, the entry is a result; if it is absent, it is a boundary-change event.

**When to use each mode:**

- Use `lineage=false` when you want administratively clean data—i.e. comparing results across elections for a seat that has not changed or been renamed.
- Use `lineage=true` when you want to trace a seat's full history back to 1955, following its dominant ancestor through every redelineation. This is what our Seats page uses to show an unbroken history even when seats were renamed or had their boundaries changed.
